## Context

RAG 问答链路（用户提问 → 检索 → 重排序 → 生成）在服务端缺少 INFO 级别的可观测性日志。当前仅有的几个 info 日志覆盖了边缘分支（simple 路由、短查询、无结果），但 happy path（检索命中 → 重排序 → LLM 生成完成）全程没有 info 日志。排查线上问题时无法通过日志追踪请求在每个阶段的状态和耗时。

本次设计延续文档上传链路日志补全的完成工作，目标一致：让 INFO 日志覆盖链路的每个节点，通过一条 trace_id 可串联整个请求的生命周期。

本变更涉及两条执行路径：
- 路径 A：SSE 流式路径（`api/chat.py:_stream_rag_response`）—— 前端用户聊天的主路径
- 路径 B：同步路径（`rag/chain.py:chat_with_citations`）—— RAGAS 评估和 GET /api/chat 使用的路径

## Goals / Non-Goals

**Goals:**

- 请求入口记录：查询文本、session_id、kb_id、路由决策（query_router 的分类结果）
- 检索阶段记录：搜索模式（hybrid/dense）、命中结果数、检索耗时
- 重排序阶段记录：输入结果数、输出上下文数、最高得分、重排序耗时
- 生成阶段记录：首 token 延迟、生成完成总耗时、token 用量（prompt/completion/total）、字符数
- 两段计时覆盖四条链路点（检索 / 重排序 / 生成 / 总计）
- 会话持久化成功后记录确认日志
- 所有改动为纯加法，不改业务逻辑

**Non-Goals:**

- 不引入新的外部依赖
- 不修改现有日志格式或级别
- 不改变 API 接口返回值
- 不做查询文本脱敏处理（原样记录）
- 不新增测试（纯日志加法无需测逻辑）
- 不处理文档上传链路的日志（已在之前完成）

## Decisions

### 1. 入口日志位置

**决定：** 日志在 `await` 之后的主协程中记录，不在线程池内。

**理由：** Loguru 的 trace_id patcher 通过 ContextVar 读取当前 trace_id。线程池内的同步调用会丢失 trace_id 上下文。路径 A 的 `_stream_rag_response` 是 async 函数，在其中直接加 `logger.info` 可保持 trace_id 完整。

### 2. 计时粒度

**决定：** 使用 `time.perf_counter()` 在主协程/主线程中四点打点（t0~t3），分别测量检索、重排序、生成的耗时。不在各子函数内部单独埋计时（不污染返回值和接口签名）。

**理由：** 三点计时的数据在各自的 `await` 返回后已有，不需要额外计算。将计时和日志集中在一处易于维护，也避免跨函数的计时变量传递。

### 3. 查询文本记录

**决定：** 原样记录用户查询原文，不做截断或脱敏。

**考虑过：** 截断前 50 字符。但排查问题时完整查询更有利于重现问题，且日志轮转保留 7 天，访问控制由基础设施保障。

### 4. 双路径重复日志

**决定：** 两条路径分别加各自的入口/计时/完成日志，底层函数（`search`、`rerank_results`、`stream_answer`）内的日志两条路径都会触发。

**理由：** 路径 A（SSE）和路径 B（同步）的入口点不同、调用堆栈不同、计时上下文不同。底层函数日志（检索模式、重排序结果数、生成完成）共享无问题——它们是同一份代码。

### 5. SSE status 与日志的关系

**决定：** 服务端日志与 SSE status 事件保持独立。SSE status 推给前端做 UI 提示（"正在检索..."），服务端日志记录量化数据（"检索耗时 0.3s，命中 15 条"），两者不冲突不重复。

## Risks / Trade-offs

- **日志量增加**：每个 RAG 请求增加约 3~5 行 INFO 日志。按目前流量预估日志存储增加 <10%，在可接受范围内。
- **查询原文落盘**：用户查询原文进入日志文件，如果日志文件访问控制不足可能导致敏感信息泄露。**当前不处理，需要的话后续可加脱敏配置。**
- **客户端断连**：如果客户端在 SSE 流传输中断开连接，生成器会抛出 GeneratorExit，完成日志不会执行。`except Exception` 会记录异常信息，这是预期行为。
- **无测试覆盖**：日志是纯加法改动，不改逻辑，因此不新增测试用例。通过观察日志输出来验证。
